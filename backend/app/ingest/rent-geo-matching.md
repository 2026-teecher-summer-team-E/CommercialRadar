# R-ONE 임대료 매칭 — 지리(좌표) 보완

R-ONE 임대료를 서울 상권에 붙이는 **이름 매칭**에, 좌표(거리)를 **보완 신호**로 얹어
이름만으로는 못 붙던 케이스를 확신 있게 구제한다. 이름 매칭은 그대로 1차로 두고,
좌표는 **"이름 실패/동점"에서 확신 게이트를 통과할 때만** 작동해 기존 매칭은 회귀시키지 않는다.

## 배경 — 왜 좌표인가

R-ONE(부동산원)은 상권을 **이름(CLS_NM)** 과 계층 경로(`서울>권역>상권명`)로만 준다. 좌표가 없다.
그래서 서울 상권 DB에 붙이려면 상권명을 맞대는 수밖에 없고, 표기 차이(광화문 vs 광화문역),
복합명(신촌/이대), 도로명(역삼대로) 때문에 완전일치는 28%(20/72)에 불과했다. 단계적 이름 매칭
(완전일치 → 부분포함 → 복합명 분해 → 트라이그램 유사도)으로 96%(69/72)까지 올렸지만,
**이름으로는 신호가 없는 3곳**(도산대로·테헤란로·숙명여대)은 의도적으로 비워뒀다(`MANUAL_MAP` 빈 리스트).

핵심 관찰: 이 3곳은 "이름"이라는 신호가 부족했을 뿐, **위치라는 신호는 있었다.** 서울 상권은
polygon geometry가 있어 centroid를 뽑을 수 있고, R-ONE 이름 상당수는 도로명·역명이라 지오코딩된다.

## 좌표 검증 (카카오 지오코딩 + PostGIS `::geography`)

R-ONE 이름을 카카오 로컬 검색으로 지오코딩해, 서울 상권 centroid와 미터 거리를 측정했다.

| R-ONE 이름 | 최근접 서울 상권 | 거리 | 판정 |
|---|---|---|---|
| 강남대로 | 강남역 | 304m | 정상매칭 확인(이미 MANUAL) |
| 역삼대로 | 역삼역 | 222m | 이름 fuzzy 정상, 좌표로 재확인 |
| 서초대로 | 서초역 253m / 서초중 368m | 253m | 최근접 명확 |
| **숙명여대** | **숙대입구 309m** / 배문고 376m | 309m | ★ 이름 무신호 → 좌표로 구제 (Δ67) |
| 도산대로 | 학동사거리 187m / 도산공원 342m | 187m | 경계선 — 사람 결정(MANUAL) 존중, 보류 |
| 테헤란로 | 강남마이스 270m / 삼성역 299m | ~270m | 애매(Δ29) → 게이트가 거부 |
| 신촌(단독어) | 봉원사 747m | 747m | ✗ 좌표 부정확 → 이름-토큰이 우수 |

**결론:** 좌표는 대체가 아니라 **보완**이다. 정상 매칭은 130~305m로 지리적으로도 옳음을 뒷받침하고,
이름 무신호 케이스(숙명여대→숙대입구)를 구제하며, 짧고 모호한 단독어(신촌)는 이름 매칭이 더 낫다.

## 설계

- **작동 시점**: `match_district_ids`가 이름(2~4단계 + fuzzy 5a/5b)으로 매칭에 성공하면 좌표는 **관여하지 않음**.
  이름이 실패하거나 트라이그램 동점일 때만 좌표가 개입 → **성공 매칭 불변(회귀 0)**.
- **확신 게이트** (`rent_transformer.py`):
  - 절대 거리 `≤ GEO_MAX_DISTANCE_M (400m)`
  - AND 최근접이 차근접보다 `≥ GEO_MARGIN_M (50m)` 더 가까움 (모호하면 거부).
  - 검증: 숙명여대(309m, Δ67) 통과 / 테헤란로(270m, Δ29) 거부 / 도산대로(187m, Δ155) 통과.
- **두 가지 티어** (fuzzy 폴백 내부):
  - **5b 동점 해소**: 트라이그램 최고점이 복수(동점)면, 좌표 최근접으로 하나 선택(게이트 통과 시).
  - **5c 지리 구제**: 이름으로 전혀 못 붙으면, 같은 시도 상권 전체에서 좌표 최근접을 게이트 하에 매칭.
- **순수성 유지**: 트랜스포머는 네트워크 없음. 지오코딩은 **오프라인 1회** (`scripts/geocode_reb.py`, 카카오)로
  `backend/app/ingest/data/reb_coords.json`을 생성하고, 런타임은 그 정적 파일만 읽는다.
- **best-effort**: `reb_coords.json`이 없거나 좌표 미보유 이름이면 좌표 보완은 자동 비활성화 → 이름 매칭만으로 동작.
- **MANUAL_MAP 존중**: 사람이 명시적으로 스킵한 항목(빈 리스트)은 좌표가 덮어쓰지 않는다.
  숙명여대만 MANUAL에서 제외해 5c로 넘겼고, 도산대로·테헤란로는 그대로 뒀다.

## 결과

- **실 DB 검증**: 순 신규 매칭은 **숙명여대 → 숙대입구(id 77)** 하나. 나머지는 전부 동일.
  → 자동 매칭 96%(69/72) → **~97%(70/72)**, centroid 309m라는 근거와 함께.
- **테스트**: 전체 215개 통과(회귀 0). rent 매칭 테스트 29 → 36 (+7: haversine, 동점 해소, 지리 구제,
  게이트 거부(애매/원거리), 좌표 없을 때 무영향, end-to-end).

## 관련 파일

| 파일 | 역할 |
|---|---|
| `transformers/rent_transformer.py` | `_haversine_m`·`_nearest_within_gate`, fuzzy 5b 동점 해소 + 5c 지리 구제, 게이트 상수 |
| `loaders/resolver.py` | `load_district_geo_map`(centroid), `load_reb_coords`(정적 좌표) |
| `jobs.py` | `ingest_seoul_rent`에서 두 로더를 `transform_record`에 배선 |
| `data/reb_coords.json` | R-ONE명 → (lat,lng) 정적 매핑 (카카오 지오코딩 결과) |
| `../../scripts/geocode_reb.py` | 오프라인 지오코딩 도구 (`.env`의 `KAKAO_REST_KEY` 사용) |
| `tests/test_rent_matching.py` | geo 보완 테스트 7개 |

## 확장 방법

- 새 R-ONE 이름 좌표 추가: `python scripts/geocode_reb.py <이름들>` → `reb_coords.json` 검수 후 커밋.
- 임계값 조정: `rent_transformer.py`의 `GEO_MAX_DISTANCE_M` / `GEO_MARGIN_M` 한 곳에서.

## 남은 팀 결정

- **도산대로**(학동사거리 187m): geo로 열려면 `MANUAL_MAP`에서 제외 → 5c가 처리. 단 학동사거리가
  의미적으로 맞는지 검토 필요(도산공원이 더 자연스러울 수 있음).
- **테헤란로**: 긴 도로라 후보가 애매(Δ29) → 게이트가 올바르게 거부. 현행 유지 권장.
